# Классы и их атрибуты и методы:
## 1. Account - (Учетная запись): абстрактный класс
- Account — это абстрактный класс, от которого наследуются другие классы учетных записей. Он содержит атрибуты для логина, пароля и роли. Эти атрибуты будут использоваться всеми производными классами.

### Атрибуты:

 - login: строка, логин пользователя.

 - password: строка, пароль пользователя.

 - role: строка, роль пользователя как, «Администратор», «Государственный служащий», «Пользователь».

## 2. Пользователь
- User — класс, наследующий от Account, представляет пользователя системы. Этот класс включает в себя личные данные 
  пользователя.

### Атрибуты:

 - lastName, firstName, middleName: фамилия, имя и отчество.

 - citizenship: гражданство.

 - passportSeries, passportNumber: данные паспорта.

 - tin: ИНН.

 - email: электронная почта.

 - city, street, house, apartment: адрес пользователя.

## 3. Государственный служащий
- GovernmentEmployee — класс, также наследующий Account, представляющий государственного служащего. Он имеет те же 
  атрибуты, что и User, но добавляется возможность обработки заявок.

## 4. Администратор
- Administrator — класс, наследующий Account, представляющий администратора системы. Подобно GovernmentEmployee, 
  администратор может управлять данными и выполнять административные функции в системе.

## 5. Заявка
- Application — класс, представляющий заявку, поданную пользователем для получения услуги.

### Атрибуты:

 - status: статус заявки (например, «в процессе», «отклонена»).

 - submissionDate: дата подачи заявки.

 - result: результат обработки заявки.

 - completionDate: дата завершения заявки.

## 6. Услуга
- Service — класс, представляющий услугу, предоставляемую системой.

### Атрибуты:

 - name: название услуги.

 - description: описание услуги.

 - startDate: дата начала предоставления услуги.
   
 - endDate: окончания предоставления услуги.

## 7. Правило
- Rule — класс, представляющий правило, которое регулирует предоставление услуги. Это правило может включать условия, которые необходимо проверить для выполнения заявки.

### Атрибуты:

 - name: название правила.

 - description: описание правила.

 - operator: оператор для сравнения (например, «>», «=», «<»).

 - value: значение для сравнения.

 - userParameter: параметр пользователя, который участвует в проверке.

 - term: срок, связанный с правилом.

## Репозитории
- Репозитории — интерфейсы, которые обеспечивают взаимодействие с базой данных для работы с сущностями.

### UserRepository: Интерфейс для работы с пользователями.

### Методы:

- findById(id: Long): найти пользователя по ID.

- save(user: User): сохранить пользователя.

- deleteById(id: Long): удалить пользователя.

### ApplicationRepository: Интерфейс для работы с заявками.

### Методы:

 - findById(id: Long): найти заявку по ID.

 - findByStatus(status: String): найти заявки по статусу.

 - save(application: Application): сохранить заявку.

 - deleteById(id: Long): удалить заявку.

### ServiceRepository: Интерфейс для работы с услугами.

### Методы:

 - findById(id: Long): найти услугу по ID.

 - save(service: Service): сохранить услугу.

 - deleteById(id: Long): удалить услугу.

### RuleRepository: Интерфейс для работы с правилами.

### Методы:

 - findById(id: Long): найти правило по ID.

 - save(rule: Rule): сохранить правило.

 - deleteById(id: Long): удалить правило.

## Сервисный слой
### Сервисный слой управляет бизнес-логикой и взаимодействует с репозиториями.

### UserService:

### Атрибуты:

- userRepository: репозиторий для работы с пользователями.

### Методы:

 - registerUser(userData: Map): регистрация пользователя.

 - updateUser(userId: Long, updateData: Map): обновление данных пользователя.

 - getUserApplications(userId: Long): получение заявок пользователя.

### ApplicationService:

### Атрибуты:

 - applicationRepository: репозиторий для работы с заявками.

 - serviceRepository: репозиторий для работы с услугами.

### Методы:

 - createApplication(userId: Long, serviceId: Long): создание заявки.

 - cancelApplication(applicationId: Long): отмена заявки.

 - getApplicationStatus(applicationId: Long): получение статуса заявки.

### GovernmentService:

### Атрибуты:

 - applicationRepository: репозиторий для работы с заявками.

### Методы:

 - processApplication(applicationId: Long): обработка заявки.

 - updateStatus(applicationId: Long, status: String): обновление статуса заявки.

 - setResult(applicationId: Long, result: String): установка результата заявки.

### AdminService:

### Атрибуты:

 - serviceRepository: репозиторий для работы с услугами.

 - ruleRepository: репозиторий для работы с правилами.

### Методы:

 - createService(serviceData: Map): создание услуги.

 - addRule(ruleData: Map): добавление правила.

 - createRoadMap(services: List<Service>): создание дорожной карты.

## Контроллеры
### Контроллеры управляют HTTP-запросами и взаимодействуют с сервисами.

### UserController:

### Атрибуты:

 - userService: сервис для работы с пользователями.

 - applicationService: сервис для работы с заявками.

### Методы:

 - submitApplication(serviceId: Long): подача заявки.

 - getApplications(): получение заявок.

 - cancelApplication(applicationId: Long): отмена заявки.

### GovernmentController:

### Атрибуты:

 - governmentService: сервис для обработки заявок.

### Методы:

 - processApplication(applicationId: Long): обработка заявки.

 - updateStatus(applicationId: Long): обновление статуса заявки.

### AdminController:

### Атрибуты:

 - adminService: сервис для администрирования.

### Методы:

 - createService(): создание услуги.

 - addRule(): добавление правила.

 - createRoadMap(): создание дорожной карты.

## Связи между сущностями
 - User → Application: Пользователь подает заявку. Один пользователь может подать несколько заявок.

 - GovernmentEmployee → Application: Государственный служащий обрабатывает заявки. Один государственный служащий может обрабатывать несколько заявок.

 - Service → Application: Услуга включает в себя заявку. Одна услуга может быть связана с несколькими заявками.

 - Service → Rule: Услуга регулируется множеством правил.

## Связи между слоями
 - Контроллеры → Сервисы: Контроллеры зависят от сервисов, так как они управляют бизнес-логикой.

 - Сервисы → Репозитории: Сервисы зависят от репозиториев для получения и сохранения данных.

 - Сервисы → Сущности: Сервисы используют сущности, такие как User, Application, Service и Rule.

## Наследование
 - Account является абстрактным классом для User, GovernmentEmployee и Administrator.
